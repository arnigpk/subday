
# План: Сжатие и оптимизация изображений в #subFlow

## Проблема
Сейчас изображения загружаются в исходном размере (до 5МБ каждое), что замедляет отображение ленты и расходует трафик пользователей.

## Решение
Реализовать сжатие изображений на стороне клиента перед загрузкой в хранилище, а также добавить прогрессивную загрузку для мгновенного отображения.

---

## Что будет реализовано

### 1. Сжатие изображений при загрузке
- Создание утилиты `compressImage` для обработки изображений на Canvas
- Уменьшение разрешения до 1200px по ширине (достаточно для мобильных экранов)
- Конвертация в WebP формат (70-80% качества) — это уменьшает размер в 3-5 раз
- Если браузер не поддерживает WebP — используется JPEG с качеством 80%
- **Результат**: фото 3-5МБ сжимается до 100-300КБ

### 2. Создание миниатюр (thumbnails)
- Генерация маленького превью (200px) для быстрой загрузки placeholder
- Хранение URL миниатюры вместе с основным изображением
- Позволяет показать "размытое" изображение пока загружается полноразмерное

### 3. Прогрессивная загрузка в ленте
- Показ placeholder/blur пока изображение загружается
- Плавное появление изображения после полной загрузки
- Native lazy loading с атрибутом `loading="lazy"`

## Технические детали

### Новый файл: `src/utils/imageCompression.ts`
```typescript
// Функции для сжатия:
// - compressImage(file, options) - сжатие с настройками
// - createThumbnail(file) - создание миниатюры
// - Поддержка WebP с fallback на JPEG
```

### Изменения в `SubFlowCreatePost.tsx`
- Обработка изображений через `compressImage()` перед загрузкой
- Показ прогресса сжатия (индикатор)
- Загрузка сжатых версий в Storage

### Изменения в `SubFlowPost.tsx`
- Добавление состояния загрузки для изображений
- Blur-placeholder пока основное изображение загружается
- Анимация появления после загрузки

### Схема данных (без изменений БД)
Сжатые изображения будут сохраняться в тот же bucket `subflow-images`, просто с меньшим размером файлов.
