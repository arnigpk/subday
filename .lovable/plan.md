# План настройки FCM и устранения ошибки с ключом

## Что уже понятно
Сейчас проблема не в том, что секрет «не принимается» как JSON, а в том, что Google отклоняет подпись JWT при получении access token.

По свежему коду и логам видно:
- edge function `send-fcm-push` читает `FCM_SERVICE_ACCOUNT` и вручную собирает JWT для `https://oauth2.googleapis.com/token`
- лог стабильно возвращает `invalid_grant` / `Invalid JWT Signature`
- значит секрет доходит до функции, но подпись либо собирается некорректно, либо `private_key` в хранилище отличается от исходного JSON, либо ключ относится не к тому Firebase/GCP проекту

## Самые вероятные причины
1. `private_key` в секрете изменился при вставке
   - лишние кавычки вокруг всего JSON
   - двойное экранирование `\n`
   - вставка через редактор/мессенджер, который испортил слэши или переносы
   - BOM/невидимые символы

2. Ключ не от того проекта
   - фронт и service worker настроены на Firebase project `subday-group`
   - если service account выпущен не для этого проекта, FCM HTTP v1 может не сойтись по проекту/правам

3. Ручная сборка JWT в edge functions слишком хрупкая
   - похожая логика дублируется в нескольких функциях
   - даже валидный ключ может ломаться из-за нюансов нормализации/кодирования

4. Есть отдельная проблема доставки после фикса ключа
   - админская рассылка сейчас берёт токены только с `platform = 'android'`
   - web-токены сохраняются как `web`, iOS тоже может не попасть в выборку
   - это не причина `Invalid JWT Signature`, но это следующая проблема, которую надо закрыть

## Что я сделаю после одобрения
1. Добавлю безопасную диагностику `FCM_SERVICE_ACCOUNT`
   - проверю обязательные поля без вывода секрета в лог
   - выведу только безопасные признаки: `project_id`, домен `client_email`, длину ключа, факт двойного экранирования, наличие корректных PEM-маркеров
   - быстро отделю «секрет испорчен» от «код подписи ломает валидный ключ»

2. Уберу ручную и хрупкую сборку JWT
   - заменю её на более надёжный общий helper для FCM auth
   - вынесу в одно место и подключу во все функции, которые шлют push

3. Приведу все push-функции к одной схеме
   - `send-fcm-push`
   - `send-subscription-notification`
   - `geo-notify`
   - `preorder-notify`
   - одинаковая нормализация секрета, одинаковые ошибки, одинаковые логи

4. Исправлю аудиторию токенов в админской рассылке
   - уберу жёсткий фильтр только по `android`
   - рассылка будет учитывать реальные сохранённые FCM токены для web/native платформ

5. Проведу тестовую отправку и сниму свежие логи
   - проверю получение access token
   - проверю ответ FCM
   - пришлю итог: что именно было причиной и какой статус отправки сейчас

## Что может потребоваться от вас
Если после диагностики окажется, что сам секрет действительно искажается, тогда я попрошу обновить его ещё раз, но уже по точному чеклисту:
- вставлять полный JSON целиком, от `{` до `}`
- ключ должен быть service account именно проекта `subday-group`
- не вставлять через Telegram/Word/форматированный редактор
- не менять `private_key` вручную

## Технические детали
- Источник ошибки сейчас: ручной OAuth JWT flow в `supabase/functions/send-fcm-push/index.ts`
- Похожие участки есть ещё в `send-subscription-notification` и `geo-notify`
- Конфиг клиента Firebase в приложении указывает на проект `subday-group`
- В админской отправке найден отдельный фильтр `device_tokens.platform = 'android'`, из-за которого часть устройств не попадёт в рассылку даже после исправления auth
- Изменения БД не требуются
